Week in Review:
This week I implemented the snapshot saving change so clip recording is now triggered by the current user's deviation score instead of the older generic recording gate. The goal of this change was to make snapshots happen when the user's form actually drifts away from the ideal stroke position, rather than just recording a fixed set of clips during a workout. On the frontend, I updated the capture workout flow so it now builds a rolling posture model from recent landmark history, identifies the dominant rowing side, and compares the user's current arm, leg, and torso positions against the idealized position for that point in the stroke. If the arm, leg, or torso deviation rises above the configured threshold and persists across consecutive checks, the app begins recording a clip. I also kept the existing in-frame checks, cooldown behavior, and clip limits in place so the new trigger works within the current workout capture system. In addition to that, I added debug state for the deviation trigger path and verified the change with a JavaScript syntax check plus the targeted deviation and workout API test suites.

Next Week:
I would like to spend time tuning the deviation thresholds with real workout footage so the snapshot trigger is sensitive enough to catch meaningful form drift without firing too often on normal landmark noise.

After that, I'd like to work on closing this issue: https://github.com/nwisnie/Erg-lytics/issues/117


Obstacles:
The biggest obstacle this week was making the deviation trigger stable enough to use in the live capture loop. Because pose landmarks can jitter from frame to frame, I had to make sure the trigger used a rolling window and repeated checks so a clip would not start from a single noisy frame.
